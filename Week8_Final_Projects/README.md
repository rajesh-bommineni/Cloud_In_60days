# Week8 Final Set of Project

Portfolio Build (Days 55–60+)

## Goal

Build one end-to-end project that demonstrates everything you’ve learned.

## Time Commitment

2 hours per day, 5–7 days (sometimes more if you refine heavily).

## What To Expect

By the end of Week 8, you should:

- Have a complete project repo on GitHub
- Be able to walk through its architecture and decisions
- Use it on your resume, LinkedIn, and in interviews

It does not have to be perfect. It must be real, end-to-end, and documented.

Step-by-Step

1.  Choose a project scope

- Ideas and patterns:
  https://www.tech5ense.com/p/high-impact-ai-driven-portfolio-projects-for-devops

2.  Beginner to Advanced level Projects here:
    https://www.youtube.com/watch?v=X6Kw2_wZr1E&t=616s

    (check description for more details)

    Cloud Resume Challenge:
    https://cloudresumechallenge.dev/docs/the-challenge/aws/

        1. Pick one realistic idea:
        	•	Notes API
        	•	URL shortener
        	•	Simple dashboard app

        2.	Design the architecture
        	•	Use Terraform for infra
        	•	Use Docker for app container
        	•	Deploy to K8s or a VM
        	•	CI/CD via GitHub Actions/Jenkins
        	•	Monitoring via Prometheus/Grafana or CloudWatch
        	•	IAM roles, secrets, and basic security
        3. Draw it using:
        https://app.diagrams.net/

3.  Implement step by step
    • Terraform: VPC, subnet, compute cluster or VM
    • Dockerize the app
    • Kubernetes (or direct VM) deployment
    • CI/CD pipeline to build and deploy
    • Monitoring and logs wiring

4.  Document everything

- README template:
  https://github.com/othneildrew/Best-README-Template

  Include:
  • Problem statement
  • Architecture diagram
  • Tech stack
  • How to run it
  • Screenshots (optional)
  • What you would improve next.

5.  Share and iterate
    • Push to GitHub
    • Post a short write-up on LinkedIn or your newsletter
    • Use the project in interviews.

## Checklist

- You have one end-to-end project deployed
- You can explain its architecture without notes
- You have a clear README with steps and diagrams
- You can describe what you’d improve with more time

## What Won’t Work

- Copying random GitHub projects.
- Your project should tell a story — problem, design, deployment, and outcome.
